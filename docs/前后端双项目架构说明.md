# 前后端双项目架构说明

版本：`v0.1`
状态：方案草案

## 1. 目标

这个项目拆成两个独立部分：

- `前端主页`：负责视觉呈现、动效、交互和内容展示
- `API 服务`：负责提供作品、技能、经历、链接等数据

当前阶段 API 可以完全基于本地数据返回结果，不依赖真实平台抓取。

## 2. 为什么要拆分

拆成两个项目的原因：

- 前端和数据解耦
- 主页可以先专注视觉和交互
- 数据层以后可以平滑替换为 GitHub、掘金、B 站、RSS、数据库等来源
- 方便调试和维护
- 后续可独立部署

## 3. 总体结构

建议采用同仓库双项目结构：

```text
Textline/
├── web/        # React 个人主页
├── api/        # 数据接口服务
├── docs/       # 架构文档
└── README.md
```

如果暂时不拆子目录，也可以先在当前仓库内约定逻辑边界，后面再迁移。

## 4. 职责边界

### 4.1 前端主页

前端只做这些事：

- 渲染首页布局
- 展示个人信息
- 调用 API 获取动态数据
- 做视觉效果和交互动画
- 处理加载态、错误态、空状态

前端不直接关心数据从哪里来。

### 4.2 API 服务

API 只做这些事：

- 提供统一数据结构
- 聚合本地数据
- 以后可替换为外部平台数据
- 处理缓存、过滤、排序、格式化

API 不负责页面渲染。

## 5. 数据流

推荐的数据流如下：

```text
本地数据文件 / 外部平台 / 数据库
            ↓
          API 服务
            ↓
        前端主页
            ↓
      页面展示与动效
```

当前阶段建议先用本地数据文件：

- `JSON`
- `TypeScript` 常量
- `Markdown`
- 轻量数据库

## 6. API 设计原则

- 统一返回结构
- 尽量保持字段稳定
- 支持分页、排序、筛选
- 支持缓存
- 允许后期替换数据源而不改前端

## 7. 推荐接口设计

### 7.1 基础信息

`GET /api/profile`

返回个人基础信息，例如：

- 昵称
- 简介
- 职业标签
- 社交链接

### 7.2 技能信息

`GET /api/skills`

返回技能列表、熟练度、分类等信息。

### 7.3 作品信息

`GET /api/projects`

返回作品列表，建议包含：

- 标题
- 简介
- 标签
- 技术栈
- 封面图
- 预览地址
- 源码地址
- 更新时间

### 7.4 经历信息

`GET /api/experience`

返回时间线数据。

### 7.5 联系信息

`GET /api/contact`

返回邮箱、GitHub、社交账号、简历下载链接等。

## 8. 统一返回格式

建议 API 都使用类似结构：

```json
{
  "success": true,
  "data": [],
  "message": "ok",
  "timestamp": 1714800000000
}
```

这样前端处理状态会更统一。

## 9. 本地数据方案

当前 API 可以完全由本地数据驱动。

### 9.1 方案一：JSON 文件

适合：

- 快速初始化
- 数据不频繁变更
- 结构简单

优点：

- 直观
- 易维护
- 适合静态接口

缺点：

- 逻辑能力弱
- 复杂处理不方便

### 9.2 方案二：TypeScript 常量

适合：

- 数据量不大
- 需要类型约束
- 希望直接在代码里处理数据

优点：

- 类型安全
- 便于复用
- 便于做数据加工

### 9.3 方案三：Markdown / MDX

适合：

- 作品说明
- 经历说明
- 博客式内容

### 9.4 方案四：SQLite

适合后期扩展：

- 数据量变多
- 需要管理历史数据
- 需要缓存外部平台结果

## 10. API 项目目录建议

```text
api/
├── src/
│   ├── data/
│   ├── routes/
│   ├── services/
│   ├── utils/
│   ├── types/
│   └── index.ts
├── package.json
└── tsconfig.json
```

### 10.1 `data`

本地静态数据源。

### 10.2 `routes`

HTTP 路由定义。

### 10.3 `services`

数据读取、筛选、排序、聚合逻辑。

### 10.4 `types`

接口类型定义。

## 11. 前端项目目录建议

```text
web/
├── src/
│   ├── components/
│   ├── sections/
│   ├── effects/
│   ├── hooks/
│   ├── services/
│   ├── styles/
│   ├── types/
│   └── App.tsx
├── package.json
└── vite.config.ts
```

### 11.1 `services`

专门放请求 API 的封装。

### 11.2 `sections`

首页各大板块。

### 11.3 `effects`

动效层，例如粒子、物理、3D。

## 12. 前端如何消费 API

前端建议只通过一个统一请求层访问数据，例如：

- `axios` 实例
- 请求拦截器
- 统一错误处理
- 统一环境变量配置

页面中不要直接写散乱的 `fetch`。

## 13. 开发顺序建议

建议按照下面顺序推进：

1. 先定义数据结构
2. 再写 API 返回本地数据
3. 再让前端接入 API
4. 再做首页展示
5. 最后叠加动效和外部平台数据源

## 14. 后续可扩展方向

后面如果要接真实平台，可以在 API 层新增数据源适配器：

- `GitHubAdapter`
- `JuejinAdapter`
- `BilibiliAdapter`
- `RSSAdapter`
- `ManualAdapter`

这样前端不用改，只有 API 内部变更。

## 15. 当前结论

现阶段最合适的方案是：

- 前端独立做个人主页
- API 独立做数据服务
- API 先用本地数据返回结果
- 以后再逐步接入真实平台数据

这能保证你先把主页效果做出来，同时不把结构做死。
